To build a truly intelligent Knowledge Graph, you need to solve the "lost context" problem. Traditional chunking breaks a document into isolated pieces, causing the LLM to lose track of what "it" or "the system" refers to. **Contextual Retrieval** fixes this by prepending a situational summary to every chunk before it hits the embedding model or the graph extraction prompt.

Here is the rewritten digestion pipeline incorporating contextualized chunking.

---

## 1. The Contextual Digestion Pipeline

This process ensures that every node and edge extracted is grounded in the document’s global intent, not just a localized snippet of text.

### Stage 1: Global Document Analysis (Python/LLM)
Before splitting the file, the system generates a "Global Context" string.
* **Action:** The LLM reads the first few thousand tokens (or a sampled summary) of the document.
* **Output:** A 200–300 word summary describing the document's purpose, main entities, and core arguments.

### Stage 2: Contextualized Chunking (Rust + Python)
Instead of storing raw text chunks, we create "Enriched Chunks."
1.  **Partitioning:** Rust's `text-splitter` crate divides the document into manageable segments (e.g., 800 tokens).
2.  **Contextual Synthesis:** For each chunk, a small LLM call (like GPT-4o-mini or a local Llama-3-8B) generates a "Situational Prefix."
    * **Prompt:** *"Given this document summary: [Global Context], write a brief 2-sentence prefix explaining what this specific chunk discusses in relation to the whole."*
3.  **The Result:** > **[Prefix]:** This chunk is part of a technical manual for the 'Medibotx' platform. It specifically details the Rust-based encryption layer used for patient data.
    >
    > **[Raw Text]:** The system implements AES-256 for all at-rest storage...



### Stage 3: Graph Extraction (The "Grounded" Triplets)
Now, the "Extraction LLM" receives the **Enriched Chunk**. Because the context is present, the LLM can perform **Co-reference Resolution** automatically.
* **Without Context:** The LLM sees "The system uses AES" and creates a node called "The System."
* **With Context:** The LLM knows "The system" refers to "Medibotx Encryption Layer" and creates a high-fidelity node.

## 2. Technical Implementation in LanceDB

To handle this efficiently, your LanceDB schema needs to store both the raw and the contextualized versions of the data.

| Column | Data Type | Purpose |
| :--- | :--- | :--- |
| `raw_content` | String | The original text (for UI display). |
| `contextual_content` | String | Prefix + Raw Text (used for Graph Extraction and Vector Search). |
| `vector` | FixedSizeList(F32) | **Embedding of the contextual_content**. |
| `doc_summary` | String | The global summary used to generate the context. |

### Rust-Side Optimization
Since generating context for 100 chunks requires 100 LLM calls, use Rust’s **`tokio::task::JoinSet`** to manage these requests in parallel. While Python handles the API calls to the LLM, Rust should stream the results directly into an Arrow `RecordBatch` to be persisted in LanceDB.

## 3. Benefits for Knowledge Graph Navigation

Integrating contextual retrieval transforms how users interact with the graph:

* **Higher Precision Filtering:** When a user filters for "Encryption," the contextual prefix ensures that chunks that don't explicitly mention the word but are *about* the topic (e.g., "AES-256 implementation") are still retrieved.
* **Cleaner Nodes:** You avoid the "Entity Explosion" where "The App," "The Software," and "The Tool" are all extracted as separate nodes. The context forces the LLM to use the canonical name (e.g., "SWClinic") across all chunks.
* **Improved Hybrid Search:** When searching the graph, the vector search is performed on the *contextualized* text. This significantly reduces "false positives" where a chunk contains the search term but in an irrelevant context.
